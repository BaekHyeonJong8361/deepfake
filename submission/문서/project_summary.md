## 2. 자체 수집 SNS OOD 데이터셋 ← 핵심 기여

### 왜 직접 수집했나?
기존 공개 딥페이크 데이터셋(FF++, DFDC, WildDF)은 모두 **고화질 원본 영상** 기준.  
실제 SNS 환경은 다르다:
- 틱톡/인스타그램에서 영상 시청 → **맥 화면녹화**(화면 기록 .mov)
- 화면녹화 과정에서 압축 블러, 해상도 저하, 인코딩 아티팩트 발생
- 기존 탐지기가 이 도메인에서 AUC 0.41로 붕괴 → **기존 벤치마크로는 우리 문제 측정 불가**

### 수집 방법
- **플랫폼**: SNS (틱톡/인스타그램 계열)
- **방식**: 맥북 화면녹화 (`화면 기록 YYYY-MM-DD 시간.mov`)
- **가짜 영상**: SNS에 유포된 딥페이크 영상 직접 수집 후 녹화
- **진짜 영상**: 동일 환경(SNS 화면녹화)에서 실제 인물 영상 수집
- **라벨링**: 팀이 직접 판별 (영상 출처 + 육안 검토)

### 수집 3단계

| 수집 차수 | 경로 | Real | Fake | 수집일 |
|---------|------|------|------|--------|
| 1차 | `final/ood_final_eval/` | 12 | 16 | 2026-05-12 |
| 2차 | `test_inputs/` | 7 | 5 | 2026-05-14 |
| 3차 | `final/ood_final_eval_v2/real_new, fake_new` | 51 | 49 | 2026-05-20 |
| **합계** | | **70** | **70** | **140개** |

### 왜 이 데이터셋이 중요한가
1. **도메인 갭 측정 가능**: 공개 데이터로 학습한 모델이 SNS에서 실제로 얼마나 무너지는지 정량화
2. **실제 배포 환경 재현**: 화면녹화 → 압축 → 블러 파이프라인이 실사용 시나리오와 동일
3. **독자적 벤치마크**: 이 평가셋 없이는 v24가 v20보다 얼마나 좋은지 측정 자체가 불가능

### 수집 시 어려움 / 한계
- 라벨링을 팀이 수동으로 진행 → 규모 확장 한계
- 특정 플랫폼(앱) 기반 딥페이크에 편향될 수 있음
- 화면녹화 기기/환경 통일 안 됨 (해상도, 밝기 편차)

## 3-4. 핵심 아키텍처 상세: F3NetLiteV20

### 왜 RGB만으론 부족한가
SNS 화면녹화 딥페이크는 육안으로 거의 구별 불가. 하지만 **GAN/Diffusion 합성 과정에서 생기는 주파수 이상**은 사람 눈에 안 보여도 FFT 분석으로 포착 가능.  
→ RGB(시각적 이상) + 주파수(숨겨진 합성 흔적) 이중 분석이 핵심 전략.

### 구조
```
입력 (299×299 RGB)
       ├─► RGB Branch (Xception backbone, 2048-d 특징)
       │
       └─► FAD (Frequency-domain Analysis)
               ├─ FFT로 주파수 변환
               ├─ num_bands=3 구간으로 나눔 (저/중/고주파)
               └─► Freq Branch (Xception backbone, 2048-d 특징)

RGB 특징 + Freq 특징
       └─► SEBlockFusion
               ├─ BatchNorm1d (각 branch 정규화)
               ├─ Channel Attention (SE Block: 어느 특징이 중요한지 가중치)
               └─► fc_head Linear(4096→1) → sigmoid → 딥페이크 확률
```
- 총 파라미터: ~44M  
- 주파수 분석(FAD): `add_on_filters` 학습 가능 파라미터로 주파수 대역 경계를 데이터에서 자동 최적화

### 2-Stage Cascade 구조
```
입력 영상
  ↓
[RYZE 모드] — v15+v16+v17 앙상블 (Xception 기반)
  → 고전 딥페이크 (FF++/DFDC 계열) 탐지. AUC 0.999
  → SNS 화면녹화엔 미사용

[SHEN 모드] — v24+v25+v29 앙상블 (F3NetLiteV20 기반)
  → SNS 특화 탐지. OOD AUC 0.884
  → 실제 데모의 핵심 모드
```
사용자가 탐지 목적에 따라 모드 선택 (LEE_SIN/SHEN/RAMMUS = 민감도 조절).

## 3-5. 얼굴 검출 파이프라인

### Mediapipe 선택 이유
- **경량·실시간**: 서버 추론 중 CPU에서도 빠름
- **SNS 영상 적합성**: 정면/측면 얼굴 모두 검출, 작은 얼굴도 잡음
- OpenCV Haar Cascade 대비 정확도 우수

### 파이프라인
```
프레임 추출 (FRAME_STEP=30, MAX_FRAMES=5)
       ↓
Mediapipe FaceDetection (model_selection=1, min_confidence=0.5)
       ↓
Bounding Box → CROP_SCALE=1.0 (얼굴 딱 맞게)
       ↓
Resize 299×299 → ImageNet 정규화
       ↓
F3NetLiteV20 추론 → sigmoid 확률
       ↓
5프레임 mean → 영상 최종 점수
```

### CROP_SCALE 실험 (5/21)
| CROP_SCALE | AUC | 특징 |
|-----------|-----|------|
| 0.8 (타이트) | 하락 | 오탐 증가 (배경 포함 안 되지만 얼굴 일부 잘림) |
| **1.0** | **0.884** | **최적** |
| 1.3 (여유) | 하락 | 배경 노이즈 포함 |

## 3-6. GradCAM / XAI 설명 기능

### 왜 넣었나
딥페이크 탐지 결과만 보여주면 "왜 가짜인가"를 설명 못 함. 발표·데모에서 신뢰도 확보를 위해 **어느 부위가 의심스러웠는지 시각화** 필요.

### 구현 방식
```
GradCAM (RGB branch의 마지막 BN layer 기준)
  + GradCAM (Freq branch 기준)
  → 두 heatmap 합산 → 얼굴 위 오버레이

SEBlockFusion gate 값 분석
  → RGB vs 주파수 기여도 비율 출력
  (예: "시각적 이상 60%, 주파수 이상 40%")

3×3 그리드 분석
  → 얼굴을 9개 영역으로 나눠 어느 부위가 가장 의심스러운지
  (예: "Forehead 0.82, Nose 0.61, Jaw 0.43")

한국어 설명 자동 생성
  → "입 주변 및 턱선에서 조작 흔적이 집중적으로 감지되었습니다."
```

## 4. 실패한 접근법 (중요 교훈)

### [실패 1] v33 — Backbone Frozen + SNS 5x 오버샘플링
- **설정**: F3NetLite backbone frozen, SNS만 5x 오버샘플링
- **결과**: val AUC 0.9998 (in-dist 완벽) → **OOD AUC 0.586 (붕괴)**
- **원인**: SNS 분포에 catastrophic overfit → 일반화 능력 소멸
- **교훈**: in-distribution 완벽 = OOD 성능 보장 안 됨

### [실패 2] v34/v35 — DINOv2 Frozen
- **v34**: DINOv2-large (304M params), 264K만 학습 (0.087%)
  - val AUC 0.9981, OOD AUC **0.667**
- **v35**: DINOv2 + 243K 대규모 데이터셋
  - val AUC 0.906, OOD AUC **0.589**
- **원인**: 학습 가능 파라미터 0.09% → SNS 도메인 적응 불가
- **교훈**: 대형 모델도 frozen이면 도메인 갭 극복 못 함

### [실패 3] v28/v30/v32 — 아키텍처 다변화 시도

| 버전 | 시도 | 가설 | 결과 |
|------|------|------|------|
| v28 | DINOv2(768d) + Xception(2048d) Fusion | Meta DINOv2로 OOD 일반화 폭증할 것 | AUC 0.66 — 차원 불균형 + 학습 파라미터 부족 |
| v30 | EfficientNet-B4 단일 백본 | 복잡한 듀얼 브랜치 버리고 SOTA 단일 백본으로 | 기존 v24/v25 앙상블 못 넘음 |

## 6. 주요 트러블슈팅

### [버그 1] MEDIAN → MEAN 집계 불일치 ★★★
- **증상**: 서버 결과가 eval AUC와 다름. fake 영상이 REAL로 판정됨
- **원인**: `inference_server_v7.py`의 `ensemble_decide()`가 `np.median()` 사용
  → 평가 스크립트는 `np.mean()` 사용 → 프로토콜 불일치
- **예시 수치**: 프레임 점수 [0.001, 0.001, 0.753, 0.685, 0.001]
  → median=0.001 (REAL 판정) vs mean=0.288 (FAKE 판정)
- **수정**: `np.median` → `np.mean` (per-frame 앙상블 + per-model score 모두)
- **파일**: `inference_server_v7.py` line 315, 318

### [버그 2] v10 MODE_CONFIG 전면 오류
- **증상**: SHEN/RAMMUS/LEE_SIN 모두 v25+v36 조합 (AUC 0.851) 사용
- **수정**:
  - 모든 SNS 모드 → v24+v25+v29 (0.2/0.5/0.3) 교체
  - SHEN 임계값: 0.11 (임의) → 0.081 (sweep 실측 최적값)
  - RYZE v17 경로: v19 파일 오참조 → `freq_only_best_v17_sns_aug.pth`
  - MAX_FRAMES: 20 → 5 (first5 프로토콜)
  - GPU: cuda:0 → cuda:1 (학과서버 규정)

### [버그 3] GPU 규정 위반
- **규칙**: 학과 서버 공유, **반드시 GPU 1번만 사용** (`cuda:1`)
- **반복 발생**: 학습 스크립트, 추론 서버 여러 곳에서 cuda:0 하드코딩
- **해결**: 모든 스크립트에 `CUDA_VISIBLE_DEVICES=1` 또는 `cuda:1` 명시

### [버그 4] cuDNN 심볼 충돌
- **증상**: `RuntimeError: CUDA error: no kernel image is available`
- **원인**: 시스템 cuDNN vs PyTorch 번들 cuDNN 충돌
- **해결**: 모든 학습 스크립트에 `torch.backends.cudnn.enabled = False`
- **환경**: venv(`~/.venv`) 내 torch 2.8 사용 필수, bare python 금지
